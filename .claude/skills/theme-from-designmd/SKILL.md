# Building a theme from a `DESIGN.md`

This skill turns a `DESIGN.md` from the [getdesign.md](https://getdesign.md/) catalog (raw source: [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md)) into a fully wired theme for the design system in this repo.

## Inputs

The user provides one of:
- **A brand slug** (`stripe`, `figma`, `vercel`, `linear.app`, …). Map it to `https://raw.githubusercontent.com/VoltAgent/awesome-design-md/main/design-md/<slug>/DESIGN.md`.
- **A raw URL** (already pointing to a DESIGN.md).
- **A local path** to a DESIGN.md file inside or outside the repo.

If the user gives a website URL like `https://getdesign.md/stripe`, derive the slug (`stripe`) and use the raw GitHub URL above — the website itself is a JS app and `WebFetch` won't return the underlying content.

## Steps

### 1. Read the canonical mapping rules

Before writing anything, read **`styles/lib/themes/README.md`** in full (especially the "Building a theme from a `DESIGN.md`" section). That file is the source of truth for which DESIGN.md keys map to which of our tokens. If a rule is missing there and you have to invent one, update the README at the same time so future runs are consistent.

### 2. Fetch the DESIGN.md

- Slug → `WebFetch` the raw GitHub URL.
- Raw URL → `WebFetch` it directly.
- Local path → `Read` it.

Ask `WebFetch` to **return the raw content verbatim** (do not let it summarize) — you need the YAML frontmatter intact and the narrative sections to disambiguate semantics.

### 3. Parse the YAML frontmatter

The frontmatter starts and ends with `---` lines. Inside, expect these top-level keys:
- `version`, `name`, `description` — metadata, ignore for token output.
- `colors` — map of color name → hex (or oklch). May include semantic names (`primary`, `success`), brand-specific names (`ink`, `canvas-cream`, `brand-dark-900`), or decorative accents (`ruby`, `magenta`).
- `typography` — map of role → `{ fontFamily, fontSize, fontWeight, lineHeight, letterSpacing, fontFeature }`.
- `rounded` — map of size name → px value.
- `spacing` — map of size name → px value.
- `components` — example component specs that reference the tokens above. Useful for understanding intent; don't emit anything for these.

The body (after the closing `---`) explains decisions in prose — read the **Colors** and **Elevation & Depth** sections to decide whether something like `ruby` is a decorative accent or a semantic error color, and whether the brand has a hero backdrop to encode as `--gradient-hero`.

### 4. Apply the mapping

Use the table from `styles/lib/themes/README.md`. Key decisions:

- **Semantic palettes**: if DESIGN.md is silent on `success` / `warning` / `error` / `info`, **do not redefine them** — let the base theme cascade.
- **Decorative colors** (`ruby`, `magenta`, `lemon` in Stripe; brand metals; etc.): emit into `--color-accent-*` slots (1, 2, 3 in order) and only if the brand uses them outside of the gradient backdrop. If a color only appears inside a gradient mesh, fold it into `--gradient-hero` instead.
- **Text ramp from one ink color**: if the DESIGN.md only gives `ink` and `ink-mute`, derive `--color-text-secondary` as a value between them in the same color space (oklch or rgb). Be explicit about derived values in a comment.
- **Typography**: set `--font-sans` once at the top of the theme if it differs. Per-role tokens go into the matching `--type-{role}-*` group. If the DESIGN.md applies a font-feature globally (e.g. `ss01`), set `--font-feature-default`.
- **Tracking values**: DESIGN.md typically gives `letterSpacing` in `px` at a specific size (e.g. `-1.4px` at `56px`). Convert to em via `px / size`, or keep the `px` value verbatim if it stays inside `--type-display-*-tracking` (it's only consumed by that specific role).
- **Sizes**: keep DESIGN.md `px` values verbatim. Tailwind 4 utilities are still available alongside the role tokens.
- **Radii**: map directly. If DESIGN.md has more than 6 radii, drop the closest duplicates rather than inventing new tokens.
- **Gradients**: write a best-effort `linear-gradient(90deg, …)` from the documented stops. Mark it with a comment if the original is an organic SVG mesh.

### 5. Write `styles/lib/themes/<brand>.css`

Format:

```css
/* ==========================================================================
 * <Brand> theme — derived from <source URL>
 * Generated by the theme-from-designmd skill.
 *
 * <one-paragraph summary from the DESIGN.md Overview section>
 * ========================================================================== */

.theme-<brand> {
  /* --- Primary ----------------------------------------------------------- */
  --color-primary-subtle:   <value>;
  --color-primary-soft:     <value>;
  --color-primary-default:  <value>;
  --color-primary-deep:     <value>;
  --color-primary-emphasis: <value>;
  --color-on-primary:       <value>;

  /* --- Surfaces ---------------------------------------------------------- */
  --color-surface-base:        <value>;
  --color-surface-raised:      <value>;
  --color-surface-inverse:     <value>;
  --color-on-surface-inverse:  <value>;

  /* --- Text -------------------------------------------------------------- */
  --color-text-default:    <value>;
  --color-text-secondary:  <value>;
  --color-text-muted:      <value>;

  /* --- Borders ----------------------------------------------------------- */
  --color-border-default:  <value>;
  --color-border-input:    <value>;

  /* --- Accents (decorative brand colors) --------------------------------- */
  --color-accent-1-subtle:  <value>;
  --color-on-accent-1:      <value>;

  /* --- Radius scale ------------------------------------------------------ */
  --radius-xs: <value>;
  --radius-sm: <value>;
  /* … */

  /* --- Typography -------------------------------------------------------- */
  --font-sans: <value>;
  --font-feature-default: <value>;

  --type-display-xxl-size:     <value>;
  --type-display-xxl-weight:   <value>;
  --type-display-xxl-tracking: <value>;
  --type-display-xxl-leading:  <value>;
  /* … */

  /* --- Decorations ------------------------------------------------------- */
  --gradient-hero: <value>;
}
```

Rules:
- **Only emit tokens that differ from the base.** Keep the file lean.
- Group overrides in the order shown above (colors → surfaces → text → borders → accents → radius → typography → decorations).
- Keep DESIGN.md values in their original notation. Hex is fine; don't convert hex → oklch (and vice versa) — nautika is hex and that's accepted convention.
- Use `/* */` block comments to call out derived values or limitations ("Best-effort approximation of the brand's organic SVG mesh.").

### 6. Wire the theme in

Add the import to `styles/index.css` (after the last existing theme import):

```css
@import "./lib/themes/<brand>.css";
```

Add the entry to `apps/storybook/.storybook/preview.ts` inside `withThemeByClassName({ themes: { … } })`:

```ts
<brand>: 'theme-<brand>',
```

Preserve the existing entries and the `defaultTheme: 'light'` line.

### 7. Report back

Print a concise summary to the user:
- **Source**: URL/slug processed.
- **Tokens mapped**: count by category (colors, typography roles, radii, decorations).
- **Tokens dropped**: anything in the DESIGN.md the mapping couldn't express (e.g. organic SVG meshes, brand-specific component shapes). Suggest a follow-up if the user wants to encode them manually.
- **Manual review flags**: derived values, semantic-vs-decorative decisions (e.g. "Treated `ruby` as `--color-accent-2-default`. If you want it as `--color-error-default` instead, override after generation.").
- **Storybook**: confirm the theme appears in the toolbar after rebuild.

## Anti-patterns

- **Don't invent semantic palettes.** If the DESIGN.md doesn't ship a success/warning/error/info palette, don't fabricate one — the base theme's defaults are good fallbacks.
- **Don't repurpose decorative colors as semantics.** Stripe's `ruby` is gradient/chart accent, not an error red. Read the narrative sections.
- **Don't add tokens to `theme.css`.** Themes only *override*. If a DESIGN.md needs a token that doesn't exist in the base, stop and ask whether to extend the base token system (a different, larger change).
- **Don't summarize the DESIGN.md when fetching it.** Ask `WebFetch` for the raw content — you need the YAML and the narrative.
- **Don't skip the README mapping table.** It's the source of truth; the steps above are just the workflow around it.
